Reference 0001

SigNoz Hackathon Strategy Map

A quick sheet for turning the rules into a project judges can understand, trust, rerun, and remember.

The scoring translation

ImpactPick a painful debugging or reliability problem. Show the before/after.
CreativityUse observability in the product loop, not just as a screenshot at the end.
Technical excellenceSmall scope, reliable path, tests or scripted demo, reproducible deployment.
Best use of SigNozTraces plus logs plus metrics plus dashboards plus alerts plus Query Builder plus MCP where possible.
User experienceOne command or clear path to value. A clean dashboard with a story.
PresentationREADME, demo video, and blog all answer: problem, architecture, telemetry, evidence, lessons.

The winning shape

Build a system that fails in an interesting way, then make SigNoz the way to understand and fix it. Judges remember cause-and-effect. A project that says "we traced an agent" is weaker than a project that says "when the agent makes a bad tool choice, this trace, log, metric, dashboard, and alert show why."

Pre-hackathon allowed work

  • Read docs and collect references.
  • Sketch architecture diagrams and demo storyboards.
  • Practice installing SigNoz with Foundry and Docker Compose.
  • Enable the optional SigNoz MCP server in a practice casting and learn the connection flow.
  • Practice with non-submission toy projects.
  • Study OpenTelemetry Collector Contrib issues and contribution workflow.

Early win before July 19

The official page includes a pre-event blog prize. The smart move is to self-host SigNoz, send toy telemetry, explore one feature deeply, and publish a real blog before July 19, 2026. This is not the final hackathon project; it is hands-on practice plus visibility.

Submission checklist

  • Repo includes casting.yaml and casting.yaml.lock.
  • Foundry path is reproducible: foundryctl gauge, foundryctl forge, and foundryctl cast are documented.
  • If using MCP, spec.mcp.spec.enabled: true is documented and the http://localhost:8000/mcp flow is explained.
  • README has a 5-minute judge path.
  • Demo shows a live or scripted failure and how SigNoz reveals it.
  • Dashboards and alerts are named for user outcomes, not internal plumbing.
  • Blog is honest: what broke, what you learned, and how telemetry changed the build.
  • AI assistant usage is disclosed.

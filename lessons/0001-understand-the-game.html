Lesson 0001

Understand the Game Before You Build

Your first win is not code. Your first win is knowing what the judges are actually rewarding.

Mission tie-in: You want to win and become visible to serious maintainers. That means building something useful, reproducible, well-instrumented, and honestly explained.

The hackathon in one sentence

You are being asked to build a project that uses or integrates with SigNoz, preferably leaning deeply on OpenTelemetry-native observability: traces, metrics, logs, dashboards, alerts, Query Builder, MCP, and reproducible Foundry deployment.

The immediate pre-event prize

The official hackathon page includes an early blog prize due before July 19, 2026. Treat that as your warm-up sprint: install SigNoz using Foundry, send toy telemetry, explore one feature deeply, and write a clear blog about what you learned. This prepares you for the main sprint without starting the final project early.

The mistake first-timers make

They build a cool app and add observability near the end. For this hackathon, observability is the product. The demo should make SigNoz feel necessary: without it, the user is confused; with it, the failure becomes explainable.

The project formula

1. PainAn agent, workflow, bot, infra task, or service behaves badly in a way real users care about.
2. SignalsYour system emits traces, logs, metrics, and cost or token-related attributes where relevant.
3. SigNoz viewA dashboard, trace exploration path, alert, or MCP workflow explains the failure.
4. ActionThe user can fix, retry, route, block, or understand the issue from the evidence.

OpenTelemetry contributor mindset

OpenTelemetry Collector Contrib maintainers respect small, well-scoped changes. The official contributing guide points contributors to local build/test commands, changelog rules, PR title rules, portability expectations, and AI-assisted contribution rules. Start by reading issues and component tests before proposing anything.

Important: Because hackathon rules allow planning before the start but say coding and design work should begin only after the hackathon starts, keep pre-work to learning, environment setup, sketches, and throwaway practice.
